# Map-Shortest-Path-Visualization-System 代码审查报告

审查范围：项目结构、地图数据建模、Dijkstra、A*、起终点选择、路径可视化、UI、工程质量与测试覆盖。  
审查原则：本轮只做分析，不修改功能代码。  
审查时间：2026-07-14。  
项目路径：`C:\Users\17629\Desktop\map-Shortest-Path-Visualization-System-master`

## 1. 项目整体结构

项目主体目录较清晰：`src/` 存放实现，`include/` 存放头文件，`data/` 存放地图数据，`tests/` 存放测试，`CMakeLists.txt` 负责构建。整体适合课程项目或小型演示系统。

主要问题是模块边界不够干净：

- 数据层、算法层、渲染层、UI 层存在一定耦合。
- `storage.c` 体量较大，包含 CSV 解析、默认数据加载、曲线路径加载、POI 处理、障碍校验相关逻辑，后续维护成本偏高。
- `renderer3d.c` 体量过大，承担渲染、交互、UI 状态、搜索、路径展示等职责，不利于定位 UI 或坐标问题。
- POI 名称、建筑名称、节点映射逻辑在数据层和 UI 层都有体现，存在未来不一致风险。
- 项目中存在旧版数据、默认数据和曲线数据并存的情况，文档与实际数据有轻微漂移。

CMake 配置整体可用，核心库启用了 `-Wall -Wextra -Wpedantic`，但 3D 可执行程序相关源文件没有完全纳入同等警告约束。CI 或本地测试默认更偏核心逻辑，3D 渲染路径没有足够自动化验证。

结论：目录结构基本合理，但模块职责边界偏粗，后续维护风险主要集中在 `storage.c` 和 `renderer3d.c`。

## 2. 图数据与地图建模

### 数据加载

当前曲线地图数据加载整体可以工作：

- `nodes.csv` 当前有 104 个节点。
- `edges_curved.csv` 当前有 161 条原始边。
- 曲线几何数据当前有 770 行，和部分说明文档中的旧数量不完全一致。
- 18 个 POI 都能映射到入口节点，并且 POI 之间当前都可达。

加载后运行时图规模约为：

- 节点：122 个。
- 边：178 条。
- 几何点：801 个。

其中 122 个节点包含原始道路节点和 POI 展示节点。POI 展示节点主要用于显示与点击，不直接作为可通行道路节点。

### 节点 ID 映射

当前已检查的 POI 入口映射是完整的，18 个 POI 都能找到对应入口节点。风险点在于：

- 外部 ID 缓冲区长度固定，较长 ID 可能被截断。
- CSV 数字解析较宽松，异常数字可能被解析为 0 或非预期值。
- `atoi`、`strtod` 类解析方式对非法输入容错过高，不利于尽早发现脏数据。

### 边连接完整性

当前道路图存在明显连通性问题：

- 主连通分量约 88 个可通行道路节点。
- `Q01/Q02/Q03` 形成独立小分量。
- 至少 13 个原始道路节点处于孤立状态或不可用于主路网。

这不会影响当前 18 个 POI 之间的可达性，但会导致地图中某些看似可以选择的位置无法作为有效路径端点，也会让部分路线被迫绕行。

### 重复边

发现一组关键重复边：

- `E017`: `M03-M04`，Central main road。
- `E054`: `M03-M04`，East gate bridge。

两条边连接同一对端点，长度相同，但几何形状和语义不同。当前加载逻辑会按无向端点合并或跳过重复几何，可能导致后加载的真实道路几何被丢弃，只保留较早的一条曲线。

这类问题在当前数据中影响有限，但如果未来支持单行道、桥梁、不同道路层级，会成为建图错误来源。

### 边权计算

当前曲线道路的边权基本按折线几何长度乘以地图比例计算，整体合理。实测误差在浮点容忍范围内。

主要风险不在边权公式，而在：

- 某些合法道路被标记为不可通行。
- 某些道路虽然在底图上明显存在，但没有进入路网。
- 曲线几何与渲染建筑轮廓不是同一套障碍模型。

### 障碍物处理

算法本身不做实时碰撞检测，完全信任边数据中的 `walkable` 标记。障碍物处理更像是离线数据质量问题，而不是 Dijkstra 或 A* 的职责。

已检查到：

- 当前可通行边基本没有命中底图校验规则中的建筑或水域。
- 但部分不可通行边实际上未明显碰撞障碍物。
- 渲染层建筑矩形和底图障碍模型不一致，可能造成“视觉上穿楼”的错觉。

### 路线异常原因判断

#### 1. 绕远路

主要属于数据问题和建图问题。

典型例子：`E027 S06-S07` 当前被标记为不可通行，但按现有像素校验规则看并未明显穿越障碍物。如果恢复这条边，部分 POI 之间路线会显著缩短。例如体育场到综合楼/第一基础实验楼附近的路径长度会从约 4040.976m 降到约 330.443m。

这说明绕远路不是算法选错，而是图中缺少一条应可通行的关键连接。

#### 2. 不经过明显更短道路

主要属于数据问题和可视化问题。

3D 模式下如果存在校园底图，系统不会突出绘制完整可通行路网。用户看到的是底图上的所有道路，但算法只能走 CSV 中标记为可通行的图边。底图道路和实际图边不一致时，就会出现“明明有更短道路却不走”的体验。

#### 3. 穿越建筑或湖泊

需要分情况：

- 如果实际图边穿越障碍物，是数据问题。
- 如果图边没有穿越底图障碍，但渲染层用较大的建筑矩形覆盖路线，是可视化模型不一致。
- 如果路线线段没有使用曲线几何，而是端点直连，则是可视化问题。

当前更主要的问题是数据模型和渲染模型不一致，而不是算法问题。

## 3. Dijkstra 算法检查

Dijkstra 实现整体正确，适用于当前非负边权图。

检查结果：

- 最短距离数组更新逻辑正确。
- relax 操作逻辑正确。
- visited 处理逻辑正确。
- 路径回溯逻辑基本正确。
- 对不可达节点可以返回无路径。

但存在以下问题：

- 当前没有使用优先队列，而是 O(V²) 线性扫描最小距离节点。当前 128 节点规模下问题不大，但不符合“优先队列实现”的要求。
- 起点等于终点时返回 `INVALID_ARGUMENT`，从路径算法语义看更合理的结果应是距离 0、路径包含起点。
- 空图、不可达节点可以处理，但测试覆盖仍可加强。
- 节点容量固定为 128，当前运行时已达到 122 个节点，离上限很近。

结论：当前 Dijkstra 算法核心正确，主要问题是边界语义和扩展性，不是导致绕远路的根因。

## 4. A* 算法检查

A* 实现整体正确，当前数据下与 Dijkstra 结果一致。

检查结果：

- `g(n)` 表示起点到当前节点的实际代价，逻辑正确。
- `h(n)` 使用坐标欧氏距离乘以比例系数，方向正确。
- `f(n) = g(n) + h(n)` 计算正确。
- 当前启发函数在现有数据中满足可采纳性，不会高估真实路径。
- 当前比例系数和边权单位基本一致。
- 对 18 个 POI 的两两路径比较，A* 与 Dijkstra 距离一致。

需要注意：

- A* 和 Dijkstra 在最短距离上应该一致。
- 在存在多条等长最短路时，两者返回的具体节点序列可以不同。
- 如果未来边权不再基于地图几何长度，或者加入道路等级、拥堵、惩罚项，当前启发函数可能不再可采纳，需要重新设计。

### 建议测试案例

- 当前旧地图固定路径：例如 `0 -> 40`、`13 -> 39`，比较 Dijkstra 和 A* 距离。
- 曲线地图 18 个 POI 两两组合，共 153 组，要求距离误差小于 `1e-6`。
- 不可达节点，例如孤立节点到主路网节点，两者都应返回无路径。
- 起点等于终点，期望距离 0。
- 等权菱形图，允许路径不同，但距离必须相同。
- 坐标重复但边权不同的图，验证启发函数不会破坏正确性。
- 全部可达道路节点的 all-pairs 比较。
- 节点数达到 128 和超过 128 的容量边界测试。

## 5. 起点/终点选择系统

当前 3D 鼠标选择逻辑只检测 POI 展示节点的盒体碰撞，不是“点击地图后找最近道路节点”。

因此会出现：

- 点击道路交叉口不能作为起点。
- 点击道路旁边空地不能选择最近节点。
- 点击非 POI 建筑或普通节点无效。
- 部分视觉上明显的位置不能作为起点或终点。

地点搜索方面，18 个 POI 的名称映射当前完整，支持英文名、别名或中文显示文本的部分匹配。但它不是通用建筑检索系统：

- 搜索结果只取第一个匹配项。
- 没有候选列表。
- 起点和终点选择逻辑不够显式。
- 任意建筑名称到节点 ID 的映射并不完整。

结论：选择系统当前更像“POI 选择”，不是完整的地图选点系统。

## 6. 路径可视化

曲线几何渲染总体正确，坐标转换和地图平面位置基本一致。路径线宽、道路线宽在当前视角下可接受。

主要问题：

- 有校园底图时，系统不绘制完整可通行路网，用户无法判断算法实际可走的图。
- 路径和建筑的渲染模型不完全一致，可能出现视觉上穿越建筑。
- POI 标记点与真实入口节点距离有时较远，路径从入口开始，但用户看到的是建筑中心或 POI 标记。
- POI 到入口的连接关系没有清晰渲染，造成路径起点/终点视觉跳变。
- 如果某条路径边没有几何数据，3D 最终路径可能跳过该边，缺少直线 fallback。
- 控制台字符地图分辨率很低，容易产生穿越障碍物的视觉误判。

结论：当前可视化对演示可用，但解释性不足。路线“看起来不贴路”多数是图边显示不足、POI 入口未显示、障碍模型不一致造成的。

## 7. UI 功能检查

### 地图缩放

缩放范围和坐标换算基本正确。风险是缩放后的贴图和文字可能不够清晰，且未做 DPI 或窗口尺寸自适应。

### 地图拖动

拖动逻辑基本可用，左键点击和拖动有阈值区分，中键可用于相机拖动。整体没有明显功能性错误。

### 地点搜索

搜索可用，但属于简单匹配：

- 只返回第一个匹配项。
- 无候选列表。
- 无模糊排序。
- 搜索目标范围有限。

### 算法切换

存在状态一致性问题：切换算法后，旧路径结果仍可能显示在界面上，用户容易误以为当前路径是新算法计算结果。应在切换算法时标记路径过期，或自动重新计算。

### 重置功能

3D UI 中的重置按钮主要重置地图视图，不完全清除起点、终点、路径、搜索状态。功能语义不够统一。

### 最大化适配

当前窗口尺寸偏固定，没有完整的最大化、可变窗口和响应式布局支持。界面最大化适配不足。

## 8. 工程质量

### 内存管理

当前没有发现明显 malloc/free 不匹配。路径结果对象有创建和销毁逻辑，raylib 资源也有释放路径。

### 指针安全

整体指针使用较直接，存在一些边界风险：

- 字体码点加载失败时的空指针处理不足。
- 文件解析异常时部分路径可能留下半初始化数据。
- 某些保存函数没有充分检查写入失败。

### 文件打开失败处理

多数文件打开失败有处理，整体可接受。但 CSV 内容格式错误时，错误定位和失败信息不够强。

### 数组越界风险

图容量固定：

- 最大节点数 128。
- 当前运行时约 122 个节点。

这意味着稍微增加 POI 或道路节点就可能触发容量限制。虽然添加函数有容量检查，但项目扩展空间很小。

### 编译警告

核心库启用了较严格警告，当前构建测试通过。但 3D 相关源文件没有完全覆盖同等 warning 策略，建议纳入。

### 其他风险

- `NaN` 边权可能绕过 `weight <= 0` 检查。
- CSV 解析不支持引号、逗号字段、空字段等真实 CSV 场景。
- `storage_save_path` 没有充分检查 `fprintf` 和 `fclose` 的失败。
- `FetchContent` 下载 raylib 未固定哈希，构建可复现性一般。

## 优化优先级

### P0 必须修，否则项目不可用

当前没有确认到会导致项目完全不可用的 P0 问题。项目可以构建、测试通过，并能完成核心寻路演示。

### P1 影响展示效果

- 复核并修正 `E027 S06-S07` 等疑似误禁用道路。
- 输出并修复孤立节点、小连通分量、缺失道路连接。
- 在地图上明确显示算法实际可走路网。
- 统一障碍物数据模型和渲染建筑模型，减少“视觉穿楼”。
- 增加点击地图选择最近可通行节点功能。
- 算法切换后清空或重新计算路径，避免旧结果误导。
- 显示 POI 到入口节点的连接关系。
- 支持窗口最大化和基础响应式布局。

### P2 提升质量

- 将 `storage.c` 拆分为 CSV 解析、图构建、POI 加载、数据校验等模块。
- 将 `renderer3d.c` 拆分为渲染、交互、UI 状态、搜索等模块。
- 改进 CSV 严格解析和错误报告。
- 把固定 128 节点容量改为动态容量或显著提高上限。
- 增加 all-pairs 差分测试、异常 CSV 测试、ASan/UBSan。
- 将 3D 构建和 warning 纳入 CI。
- 统一 reset 语义。
- 修复起点等于终点的边界行为。
- 为缺少 geometry 的路径边增加直线渲染 fallback。

## 总结

当前项目的核心算法实现基本可靠，A* 和 Dijkstra 在现有 POI 数据上结果一致。路线异常的主要根因不是 Dijkstra 或 A* 算错，而是地图数据、建图规则和可视化表达之间存在不一致。

优先修复方向应放在数据质量和可视化解释性上：先让图中“哪些路可走”对用户可见，再修正误禁用道路和孤立连接，最后再处理模块拆分、解析健壮性和自动化测试覆盖。

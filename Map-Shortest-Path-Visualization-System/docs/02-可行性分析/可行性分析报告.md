# 地图最短路径可视化系统可行性分析报告

## 1. 选题背景

最短路径是数据结构、图论和人工智能搜索中的典型问题。普通控制台程序只能给出路径结果，不利于观察节点访问顺序和不同算法的搜索范围，因此本项目以可视化演示为目标，对 Dijkstra 与 A* 进行统一建模和对比。

## 2. 技术可行性

- 使用 C11 实现图、节点、边、路径结果及算法模块，课程知识覆盖明确。
- Dijkstra 与 A* 都可基于统一图接口实现，模块边界清晰。
- CMake 提供跨平台构建，CTest 用于自动运行模块测试。
- CSV 作为初期数据格式，便于人工编辑、复现实验和后续扩展。
- 当前终端可视化可验证算法过程，后续可替换为图形窗口而不修改算法接口。

## 3. 经济可行性

项目使用 CMake、Git 和常见 C 编译器等免费工具，不需要采购服务器、数据库或商业地图授权。开发与演示可在普通教学电脑上完成，成本较低。

## 4. 操作可行性

系统采用菜单式操作，用户选择算法、起点和终点后即可查看访问顺序、最短路径及总距离。示例地图随项目提供，便于教师和同学直接运行。

## 5. 进度可行性

项目可分为工程搭建、图与存储、算法实现、可视化、测试、文档和答辩七部分。四名成员职责明确，任务可并行推进，并通过统一接口进行集成。

## 6. 风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| A* 启发函数不合理 | 可能失去最优性或效率优势 | 使用欧氏距离，并保证边权不小于对应直线距离 |
| CSV 数据错误 | 地图加载失败 | 校验字段数量、节点编号和非负权重 |
| 模块接口频繁变化 | 集成返工 | 先评审头文件，接口变更同步通知全组 |
| 展示效果不足 | 答辩说服力下降 | 先完成访问顺序展示，再迭代图形界面和动画 |

## 7. 结论

本项目技术路线成熟、成本低、任务规模适中，能够覆盖课程要求并形成直观演示成果，具备实施可行性。

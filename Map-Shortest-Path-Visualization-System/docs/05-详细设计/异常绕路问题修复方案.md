# 异常绕路问题修复方案

> 实施状态（新版底图）：已完成。原 `M/S/D` 节点与 `E156` 等边属于旧版
> 1920×1356 底图，仅作为问题复盘保留。当前系统改用 1448×1086 的
> `campus_map_new.png`，路网由 `rebuild_map_data.py` 统一生成；建筑与环境学院楼
> 接入 `J70`，综合楼与一基楼使用独立入口 `P02`，两者通过东园与中央教学区道路
> 直接连通，不再进入西园宿舍路网。

## 1. 问题说明

在 3D 校园导航中，从“建筑与环境学院楼”前往“综合楼与一基楼”时，推荐路线没有使用中央区域已有道路，而是绕行西区和南侧外环。

当前地点接入关系如下：

- 建筑与环境学院楼接入路网节点 `M05`；
- 综合楼与一基楼接入路网节点 `S06`。

程序当前生成的最短路径为：

```text
M05 -> M04 -> L05 -> L04 -> L03 -> L02 -> S01
-> D08 -> D07 -> D06 -> D05 -> D09 -> D13
-> B01 -> B02 -> B03 -> B04 -> B05
-> A02 -> A03 -> S05 -> S06
```

路径总长为 `3019.1 m`，预计步行时间为 `37.7 min`。该路线虽然在现有图结构中可达，但明显不符合校园实际通行情况。

## 2. 根因

问题不在 Dijkstra 或 A* 算法，而在算法使用的路网数据。

校园底图中的白色道路只是图片内容，寻路算法不会自动识别这些道路。算法只使用 `edges_curved.csv` 中 `walkable=1` 的边；`walkable=0` 的边仅可用于数据表达或绘制，不会进入寻路邻接矩阵。

中央区域目前存在以下断点：

| 边 | 道路 | 当前状态 | 影响 |
| --- | --- | --- | --- |
| `E023` | `S02-S03` 明远湖南侧主路 | `walkable=0` | 湖南侧道路中断 |
| `E024` | `S03-S04` 明远湖南侧主路 | `walkable=0` | 湖南侧道路中断 |
| `E025` | `S04-S05` 明远湖南侧主路 | `walkable=0` | `S05/S06` 无法从西侧直接进入 |
| `E088` | `S03-Q01` 长桥西接入 | `walkable=0` | 长桥西端未接入路网 |
| `E091` | `Q03-S05` 长桥东接入 | `walkable=0` | 长桥东端未接入路网 |
| `E156` | `M05-S05` 中央教学区纵向主路 | `walkable=0` | 起点与南侧道路之间断开 |
| `E157` | `M06-S06` 中央教学区纵向主路 | `walkable=0` | 中央教学区与终点之间断开 |
| `E159` | `S06-B07` 南北主连接路 | `walkable=0` | 终点南侧连接断开 |

其中 `S05/S06` 区域当前主要通过 `A03-S05` 进入，形成单一入口。算法因此只能先到达西区或南侧外环，再经过 `A03-S05` 进入终点区域。

部分禁用边只有首尾两个几何点。如果直接把这些边的 `walkable` 从 `0` 改为 `1`，渲染和寻路会采用端点直线，可能穿过建筑或湖泊。因此不能仅修改通行标志，必须先修正道路几何。

## 3. 修复目标

修复后应同时满足以下要求：

1. 路线不得穿过湖泊或建筑；
2. 底图中真实存在的步行道路应进入寻路图；
3. `M05` 到 `S06` 不再经过西区外环；
4. `S05/S06` 区域至少具有两条独立的步行接入路线，避免单入口瓶颈；
5. 建筑与环境学院楼到综合楼与一基楼的推荐路线应控制在 `700 m` 以内；
6. Dijkstra 与 A* 应得到相同的最短距离。

## 4. 数据修复方案

### 4.1 沿真实道路重新描绘折线

以 `M05-S05` 或 `M06-S06` 为基础，沿校园底图中的真实道路中心线重新设置几何点。道路需要绕过建筑轮廓和水域，不得直接连接端点。

如果一条道路包含多个转弯或交叉口，应在 `nodes.csv` 中增加道路节点，并拆分为多条较短的边。例如：

```text
M05 -> 新增转弯节点 -> 新增路口节点 -> S05 -> S06
```

或者：

```text
M05 -> M06 -> 新增转弯节点 -> S06
```

每条边的第一个几何点必须与源节点坐标一致，最后一个几何点必须与目标节点坐标一致。

### 4.2 恢复中央区域连通性

完成几何修正并通过障碍检查后，将对应边设置为 `walkable=1`。建议至少恢复两组通道：

- 教学区至综合楼方向的中央通道；
- 湖南侧道路或长桥方向的备用通道。

这样既能消除当前绕路，也能避免任意一条边失效后再次形成大范围绕行。

### 4.3 更新距离和控制点数量

道路权重必须按照折线总长度计算：

```text
distance_px_curved = 所有相邻几何点之间的欧氏距离之和
distance_m_est_curved = distance_px_curved * 0.8
geometry_point_count = 该边在 edge_geometry_points.csv 中的点数
```

需要同步修改：

- `data/curved/nodes.csv`：新增转弯点或交叉口；
- `data/curved/edges_curved.csv`：更新边、通行状态、距离和控制点数量；
- `data/curved/edge_geometry_points.csv`：写入沿真实道路描绘的折线；
- `data/curved/campus_graph_curved.json`：如果继续将其作为数据发布文件，应同步更新；
- `data/curved/curved_graph_overlay_preview.png`：重新生成路网覆盖预览。

## 5. 实施步骤

1. 在 `campus_map_new.png` 上确定新版道路中心线和每个建筑的独立入口；
2. 沿道路转弯处记录坐标，必要时向 `nodes.csv` 添加节点；
3. 在 `edge_geometry_points.csv` 中重建相关边的有序控制点；
4. 重新计算曲线像素长度和米制距离；
5. 更新 `edges_curved.csv`，仅将通过障碍检查的边设为 `walkable=1`；
6. 接通湖南侧道路或长桥，形成第二条可用通道；
7. 运行数据验证、连通性测试和单元测试；
8. 重新启动 3D 界面，人工检查橙色路线是否贴合道路。

## 6. 验证方法

### 6.1 障碍与数据一致性检查

项目已有验证脚本，可检查几何点数量、折线长度以及道路是否接触建筑或水域：

```powershell
.\data\curved\validate_map_data.ps1
```

也可以只检查本次修改的边：

```powershell
.\data\curved\validate_map_data.ps1 -EdgeId E156,E157
```

桥梁允许跨水，但普通道路不得跨水。所有 `walkable=1` 的边都必须通过验证。

### 6.2 POI 连通性检查

```powershell
python .\data\curved\check_connectivity.py
```

该检查保证所有 POI 入口相互可达，但不能单独证明路线合理，因此还需要距离断言。

### 6.3 路线合理性测试

在 `tests/test_curved_campus.c` 中增加以下回归测试：

```text
起点：建筑与环境学院楼（M05）
终点：综合楼与一基楼（S06）
预期：能够找到路径，且总距离小于 700 m
```

同时分别执行 Dijkstra 和 A*，验证：

```text
两种算法均能找到路径；
两种算法的总距离一致；
路径不包含不可步行边；
路径不再经过西区外环节点 D07、D06、D05、D09、D13、B01、B02。
```

完成数据修改后运行：

```powershell
cmake --build build-3d --config Release
ctest --test-dir build-3d -C Release --output-on-failure
```

## 7. 验收标准

修复满足以下全部条件后方可完成：

- 数据验证脚本无建筑、水域、长度或控制点数量警告；
- 所有 POI 入口保持双向可达；
- 9 项现有测试及新增路线合理性测试全部通过；
- 建筑与环境学院楼到综合楼与一基楼的路径小于 `700 m`；
- 路线沿真实道路绘制，不穿湖、不穿建筑；
- Dijkstra 与 A* 输出一致；
- 3D 界面中橙色路线不再绕行西区外环。

## 8. 注意事项

- 不要仅修改 `walkable`，否则可能产生穿楼或穿湖的直线路径；
- 不要通过人为降低某条错误边的权重来强制算法选择它；
- 不要修改 Dijkstra 或 A* 来规避数据问题；
- 底图可见道路必须转换为图节点、可通行边和折线几何后，才能参与寻路；
- 连通性通过不等于路线合理，必须同时验证障碍、距离和典型路线。

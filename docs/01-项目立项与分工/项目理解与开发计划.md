# C语言实训项目理解与开发计划

## 【项目一句话概括】

以 3～5 人小组形式，选择一个小型信息管理系统，用 C 语言完成模块化开发、文件持久化、测试、文档和最终答辩。

老师并未规定唯一题目，可选：

- 图书管理系统
- 通讯录管理系统
- 学生综合测评系统
- 学生选课及学籍管理系统
- 火车票务管理系统
- 购物卡管理系统
- 超市收银系统

也可自拟，但必须先经老师批准。材料对“超市收银系统”描述最完整，因此它是风险最低的选择。

## 【老师的核心要求】

1. 使用 C 语言开发，推荐 Code::Blocks 或其他支持 C 的 IDE。
2. 必须采用多文件结构。
3. 每个主要模块单独放在一个 `.c` 文件中。
4. 宏、结构体、函数声明放在 `.h` 文件中，并使用头文件保护。
5. 使用结构体数组或链表保存业务数据。
6. 掌握指针、动态内存、文件读写、排序和格式化输出。
7. 数据必须能从文件加载，并在修改后重新保存，不能只存在内存中。
8. 输入必须校验，包括数字格式、范围、长度、重复数据和异常字符。
9. 系统应具有一定安全性和健壮性。
10. 需要按需求分析、总体设计、详细设计、编码、调试、测试的流程完成。
11. 代码必须按天增量提交 Git，禁止最后一次性提交全部代码。
12. 每位成员必须能够逐行解释自己的代码，并回答边界条件、复杂度和设计决策。
13. 借助 AI 调试或查询语法的代码，必须在代码头部明确标注，否则视为违规。

如果选择超市收银系统，核心功能为：

- 商品添加、查询和库存调整。
- 同一商品不得重复创建，应累加库存。
- 库存不能为负数。
- 收银时输入商品编号和数量。
- 显示购物清单、单价、数量、小计和总价。
- 禁止超卖。
- 结算后自动扣减并保存库存。

登录、角色权限、商品上下架、挂单、销售记录、月度统计、数据库等属于扩展功能。

## 【建议技术栈】

推荐采用：

- 语言：C11。
- 编译器：GCC/MinGW。
- IDE：Code::Blocks、VS Code 或 CLion。
- 构建：CMake，若课程更基础也可使用 Makefile。
- 数据存储：CSV、文本或二进制文件。
- 版本控制：Git + GitHub/GitLab。
- 图表工具：ProcessOn、draw.io。
- 测试：独立测试程序加人工黑盒测试用例。

不建议一开始使用 Web 前后端、Vue、Java、Python 或 MySQL。正式设计要求明确指向 C 语言多文件控制台程序，数据库只在扩展建议中出现。

课程表里的“前后端 API、数据库、UI 页面”带有通用 Web 项目内容，与 C 语言正式要求存在冲突，应先向老师确认是否真的适用于本班项目。

## 【开发步骤】

1. 组队、选题、确定组长和成员角色，并让老师确认题目。
2. 建立 Git 仓库，约定分支、提交信息和每日提交规则。
3. 编写可行性分析、用户角色、业务流程和功能清单。
4. 锁定第一期必做范围，不要过早添加登录、数据库等扩展功能。
5. 设计结构体、文件格式、模块关系和函数接口。
6. 建立多文件项目骨架，先实现菜单和数据文件加载。
7. 依次实现查询、添加、修改、删除或库存管理等基础功能。
8. 实现核心业务流程，例如收银、结算和库存联动。
9. 补充输入校验、重复检查、边界处理和异常恢复。
10. 编写并执行正常、异常、边界和数据持久化测试。
11. 修复 Bug，进行 Code Review，保持文档与实际代码一致。
12. 完成用户手册、测试报告、项目总结、答辩 PPT 和演示录屏。

每天只提交当天真实完成的增量，并确保所有成员能解释自己的代码。

## 【最先要写的文件】

第一批建议创建：

- `README.md`：记录项目名称、选题、成员、编译和运行方式。
- `.gitignore`：排除 `build/`、可执行文件和 IDE 临时文件。
- `CMakeLists.txt` 或 `Makefile`：建立统一编译方式。
- `docs/01-项目立项与分工/团队角色分工表`：明确每个人负责什么。
- `docs/02-可行性分析/可行性分析报告`：说明为什么选择该题目。
- `docs/03-需求分析/功能清单.xlsx`：列出模块、功能、优先级和验收条件。
- `include/common.h`：公共宏、状态码和全局约定。
- `include/models.h`：定义商品、用户、订单等结构体。
- `src/main.c`：只建立主菜单和程序入口，不急着写完整业务。
- `include/storage.h`、`src/storage.c`：统一规划文件加载和保存接口。
- `data/products.csv`：准备少量可重复测试的初始数据。

第一天不要一次性把所有模块写完，否则既不利于设计，也会违反增量提交要求。

## 【重点得分点】

材料中没有发现明确的百分制评分表。按老师反复强调的内容，优先级应为：

1. 代码真实性与可解释性：能逐行解释、能手写关键算法、能回答边界条件。
2. 每日增量提交：Git 历史必须真实连续，不能最后一次性上传。
3. 多文件与结构化设计：模块独立、头文件规范、职责清楚。
4. 核心功能正确：数据增删改查、业务联动和文件持久化必须可靠。
5. 输入与边界校验：非数字、负数、重复编号、库存不足、空文件、损坏数据都要处理。
6. 文档与实现一致：需求、流程图、详细设计、测试用例不能和代码“两张皮”。
7. 测试覆盖：正常、异常、边界、持久化、回归测试均要有记录。
8. 团队协作：分工明确、提交记录清楚、代码评审可追溯。
9. 演示稳定：准备固定演示数据及录屏备用方案。
10. 扩展功能：应在基本功能稳定后再做。

参考质量分析材料中，最常见问题是功能错误、输入长度未校验、需求不清和 UI 不一致。这些不是评分权重，但很可能是老师检查的重点。

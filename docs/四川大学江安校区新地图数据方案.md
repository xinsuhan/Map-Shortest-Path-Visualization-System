# 四川大学江安校区新地图数据方案

本文档基于新上传的四川大学江安校区高清地图，以及当前 `Map-Shortest-Path-Visualization-System` 的数据结构，整理新的底图与道路网络重建方案。

当前阶段只做分析和设计，不修改代码、不修改 CSV 数据。

## 1. 当前项目地图数据结构分析

### 1.1 `data/curved/nodes.csv`

当前字段：

```csv
id,x,y,nx,ny,name,kind
```

字段含义：

- `id`：道路节点编号，例如 `S06`、`N12`。
- `x,y`：节点在底图像素坐标系中的位置。
- `nx,ny`：归一化坐标，通常由 `x,y` 除以地图宽高得到。
- `name`：节点名称或说明。
- `kind`：当前加载逻辑要求为 `intersection`。

当前数据特点：

- 当前 `nodes.csv` 主要存储道路交叉点、道路转折点、桥梁端点等算法节点。
- POI 不是直接写在 `nodes.csv` 中，而是由 `pois.csv` 单独维护。
- 当前项目容量限制较紧：`MSP_MAX_NODES` 为 128。现有道路节点约 104 个，再加 POI 显示节点后已经接近上限。

设计影响：

- 新地图如果要更真实，应增加道路节点数量。
- 如果新增节点较多，后续需要提高 `MSP_MAX_NODES`，否则无法完整加载高精度校园路网。
- 算法本身不需要修改，主要是数据规模上限需要调整。

### 1.2 `data/curved/edges_curved.csv`

当前字段：

```csv
id,source,target,road,kind,oneway,width,distance_px,distance_m_est,distance_px_curved,distance_m_est_curved,geometry_point_count,walkable
```

字段含义：

- `id`：边编号，例如 `E027`。
- `source,target`：起点节点和终点节点。
- `road`：道路名称或区域名称。
- `kind`：道路类型，例如 `road`、`pedestrian`、`bridge`。
- `oneway`：是否单向。建议新数据统一使用 `False` 或 `True`，避免使用历史兼容值。
- `width`：道路宽度标记，例如 `wide`、`medium`、`narrow`。
- `distance_px`：源点到目标点的直线像素距离。
- `distance_m_est`：直线距离估算的米数。
- `distance_px_curved`：按曲线几何点计算的像素距离。
- `distance_m_est_curved`：曲线路径估算的米数，也是算法实际使用的权重。
- `geometry_point_count`：该边在 `edge_geometry_points.csv` 中的几何点数量。
- `walkable`：是否允许算法通行。`true` 表示属于搜索空间，`false` 表示仅显示或辅助连接。

当前数据特点：

- Dijkstra 和 A* 使用 `distance_m_est_curved` 作为路径权重。
- `walkable=false` 的边不应参与算法搜索。
- 当前加载逻辑会合并重复的无向边，重复边可能导致后出现的几何数据被忽略。

设计影响：

- 新数据必须避免重复边。
- 每条真实可走道路都应设置 `walkable=true`。
- 建筑内部、湖面、草坪穿越线、装饰性底图道路不应设置为可通行边。

### 1.3 `data/curved/edge_geometry_points.csv`

当前字段：

```csv
edge_id,source,target,road,point_order,x,y
```

字段含义：

- `edge_id`：对应 `edges_curved.csv` 中的边编号。
- `source,target`：该几何点所属边的起止节点。
- `road`：道路名称。
- `point_order`：曲线点顺序。
- `x,y`：几何点在底图上的像素坐标。

当前数据约束：

- 每条边的几何点应按 `point_order` 连续排列。
- 每条边的第一个几何点应等于 source 节点坐标。
- 每条边的最后一个几何点应等于 target 节点坐标。
- `geometry_point_count` 应与实际几何点数量一致。

设计影响：

- 新地图为透视插画，不是严格 GIS 正射图，不适合只用直线边。
- 湖边、弯路、桥梁、环形道路应使用多点曲线几何。
- 几何点数量会明显增加，建议后续提高 `MSP_MAX_GEOMETRY_POINTS`。

## 2. 新地图节点划分方案

新地图建议按“道路搜索节点”和“地点 POI”分开设计。

道路搜索节点写入 `nodes.csv`，用于算法寻路；地点 POI 写入 `pois.csv`，用于用户选择和显示。

### 2.1 教学楼节点

教学楼不应直接作为道路节点，而应使用“入口连接点”建模：

- 第一教学楼：主入口、东西侧道路接入点。
- 第二教学楼：主入口、南侧道路接入点。
- 文理楼群：靠近东校门一侧入口、连接文科楼道路入口。
- 工程训练中心：主入口、宿舍区连接点。
- 学术交流中心：湖区侧入口、中央广场侧入口。

建议节点类型：

- 建筑入口点：放在建筑外侧道路上。
- 教学区主路交叉点：放在教学楼之间的道路交汇处。
- 教学区道路转折点：用于贴合底图道路弯曲。

### 2.2 图书馆节点

图书馆位于明远湖东侧、行政楼附近，是校园核心节点。

建议设置：

- 图书馆北入口节点。
- 图书馆南入口节点。
- 图书馆西侧湖边步道节点。
- 图书馆东侧行政楼连接节点。

图书馆区域应与行政楼、明远湖环线、中心广场、教学区主轴连接。

### 2.3 体育馆和体育场节点

体育区域包括体育馆、体育运动场、体育场和周边道路。

建议设置：

- 体育馆入口节点。
- 体育运动场西侧节点。
- 体育运动场南侧节点。
- 体育运动场东侧节点。
- 连接二教、文科楼、南一路的道路节点。

注意：

- 体育场内部跑道不能被当作普通通行捷径。
- 只能沿场馆外部道路和明确出入口连接。

### 2.4 食堂节点

地图中可识别的食堂区域包括学生食堂、东园学生食堂等。

建议设置：

- 西园宿舍区食堂入口节点。
- 东园学生食堂入口节点。
- 南侧教工生活区餐饮服务节点。

食堂节点应挂接到附近道路，而不是直接连到建筑中心。

### 2.5 宿舍区节点

宿舍区应按片区网格建模。

西园宿舍区：

- 西校门入口节点。
- 宿舍区南北主路节点。
- 1舍、2舍、3舍、6舍、7舍、9舍、10舍、11舍、12舍等建筑群入口节点。
- 13舍、14舍、15舍、16舍、17舍、18舍、19舍、20舍、21舍、22舍、23舍等北侧宿舍入口节点。

东园宿舍区：

- 东园一舍、二舍、七舍、八舍入口节点。
- 东园主路交叉点。
- 东校门连接节点。

南侧教工生活区：

- 教工生活区主入口。
- 内部环路交叉点。
- 江安大道南段接入点。

### 2.6 道路交叉点

道路交叉点是新图最重要的节点类型。

建议重点布点：

- 大件路北侧边界道路。
- 长堤路西侧边界道路。
- 江安大道南段。
- 明远大道北段。
- 明远湖环湖道路。
- 教学区东西向主路。
- 体育场周边道路。
- 东校门、南门、西校门连接道路。

节点应放在道路中心线交汇处，不应放在道路边缘。

### 2.7 桥梁节点

江安河和明远湖周边桥梁必须独立建模。

建议每座桥至少设置：

- 桥西端节点。
- 桥东端节点。
- 桥中点或桥面曲线点。

桥梁应使用 `kind=bridge`，并设置 `walkable=true`。

禁止将河两岸道路直接跨水连接，除非该处明确存在桥梁。

### 2.8 湖区连接点

明远湖是校园路径绕行的主要原因之一，湖区节点必须细化。

建议设置：

- 明远湖北岸步道节点。
- 明远湖南岸步道节点。
- 明远湖西岸步道节点。
- 明远湖东岸图书馆连接点。
- 湖中岛连接点。
- 湖边广场连接点。
- 学术交流中心连接点。

湖区道路应沿岸线弯曲，不允许穿湖直连。

## 3. 建议新增节点数量

当前道路节点约 104 个。新地图建议重建为约 156 个道路节点，并配套约 30 个 POI。

建议道路节点分区如下：

| 分区 | 前缀 | 建议数量 | 说明 |
| --- | --- | ---: | --- |
| 校门与边界道路 | `GT` | 6 | 西校门、南门、东校门、北侧边界接入 |
| 西北科研区 | `NW` | 14 | 多学科交叉楼、科研院、江安研究所周边 |
| 西园宿舍区 | `WD` | 22 | 西园宿舍网格、食堂、运动场 |
| 明远湖与环湖道路 | `LK` | 24 | 环湖步道、岛、广场、湖岸转折点 |
| 桥梁 | `BR` | 12 | 江安河、明远湖桥梁两端 |
| 中央行政图书馆区 | `CA` | 16 | 行政楼、图书馆、中心广场 |
| 教学区 | `TA` | 20 | 一教、二教、文理楼、文科楼 |
| 体育区 | `SP` | 12 | 体育馆、体育运动场、体育场外环 |
| 东园与东校门区域 | `EA` | 18 | 东园宿舍、支科楼、东校门 |
| 南侧生活区 | `ST` | 12 | 教工生活区、南侧道路 |

合计建议：

- 道路节点：约 156 个。
- POI：约 30 个。
- 边：约 230 到 250 条。
- 曲线几何点：约 1500 到 2000 个。

容量影响：

- 当前 `MSP_MAX_NODES=128` 不足以承载该方案。
- 后续实施前建议将节点上限提高到至少 256。
- 当前 `MSP_MAX_GEOMETRY_POINTS=1024` 也偏小，建议提高到至少 4096。

这属于数据容量扩展，不涉及 Dijkstra 或 A* 算法逻辑修改。

## 4. 节点之间连接方式

### 4.1 主干道路

建议建立以下主干连接：

- 西校门 -> 西园宿舍区 -> 体育场 -> 南门。
- 西校门 -> 江安大道南段 -> 南门 -> 教工生活区。
- 北侧大件路 -> 西北科研区 -> 明远大道北段。
- 明远大道北段 -> 图书馆 -> 行政楼 -> 中央广场。
- 中央广场 -> 第一教学楼 -> 第二教学楼 -> 文理楼区域。
- 体育场周边 -> 体育馆 -> 东侧教学区 -> 东校门。
- 东校门 -> 东园宿舍区 -> 文科楼区域。

### 4.2 湖区道路

明远湖应采用环湖步道，而不是少量直线边。

建议连接：

- 湖西岸步道连接江安河桥。
- 湖北岸连接明远大道北段。
- 湖东岸连接图书馆和行政楼。
- 湖南岸连接学术交流中心、宋英楼和南门方向。
- 湖中岛只通过可见桥或步道连接，不得跨水直连。

### 4.3 桥梁连接

桥梁连接应独立成边：

- 江安河西北桥：连接西北科研区与明远大道北段。
- 明远湖西侧桥：连接西园宿舍区与湖区。
- 明远湖南侧长桥：连接体育场、宋英楼、教学区。
- 南门附近桥：连接南侧道路和教工生活区。

每座桥应拆成：

- 桥头道路节点。
- 桥梁边。
- 桥尾道路节点。

这样可以避免算法误判河流两岸直接连通。

### 4.4 宿舍区内部连接

宿舍区适合使用网格道路：

- 每栋宿舍入口连接到最近道路节点。
- 宿舍区主路与支路交叉点都应建节点。
- 支路之间不应穿过建筑或绿地直接连接。

### 4.5 教学区内部连接

教学区道路较密，建议按“主路 + 建筑入口支路”建模：

- 第一教学楼、第二教学楼之间设置东西向主路。
- 大礼堂、操场、教学楼之间设置南北向连接。
- 建筑入口边距离较短，权重应反映真实步行距离。

## 5. 需要特别避免的区域

### 5.1 建筑不可穿越

所有建筑轮廓应视为不可通行区域，包括：

- 教学楼。
- 图书馆。
- 行政楼。
- 体育馆。
- 宿舍楼。
- 食堂。
- 科研楼。

道路节点应放在建筑外侧入口，不应放在建筑中心。

建议对建筑轮廓做 6 到 10 像素膨胀，避免路径贴着建筑边缘穿过。

### 5.2 湖泊和河流不可穿越

明远湖、江安河不能直接连边。

只有以下情况可以跨水：

- 明确画有桥梁。
- 明确画有栈道或可通行步道。

湖岸路径应沿岸线走，不能使用直线穿湖。

建议对水体边界做 4 到 8 像素膨胀，避免曲线几何误入水面。

### 5.3 草坪不可随意穿越

地图中大量绿色区域是草坪、绿地或景观区域。

处理原则：

- 只有清晰画出的白色或灰色步道可作为 `walkable=true`。
- 草坪中没有明显道路时，不建立可通行边。
- 景观弯路可以作为步行边，但必须贴合实际路径。

### 5.4 体育场内部不可作为捷径

体育运动场和体育场内部跑道不应作为普通路径搜索空间。

可通行部分应限制在：

- 场馆外部道路。
- 明确入口。
- 场馆周边铺装道路。

### 5.5 避免重复边和覆盖问题

当前加载逻辑会合并重复的无向边，因此新数据必须避免：

- 同一对节点存在两条无向边。
- 一条真实道路被重复拆成相同 source 和 target。
- 后一条重复边试图覆盖前一条边的 geometry。

如果确实存在两条平行路，应使用不同的中间节点拆开，而不是使用相同 source 和 target。

## 7. 新 `edges_curved.csv` 设计方案

边 ID 建议按区域分段：

| ID 范围 | 区域 |
| --- | --- |
| `E001-E039` | 校园外环和主干道 |
| `E040-E079` | 西园宿舍区 |
| `E080-E119` | 明远湖和桥梁 |
| `E120-E159` | 行政图书馆中心区 |
| `E160-E199` | 教学区 |
| `E200-E229` | 体育区 |
| `E230-E259` | 东园和东校门区域 |
| `E260-E289` | 南侧教工生活区 |

示例结构：

```csv
id,source,target,road,kind,oneway,width,distance_px,distance_m_est,distance_px_curved,distance_m_est_curved,geometry_point_count,walkable
```

边设计原则：

- 所有真实可步行道路设置 `walkable=true`。
- 不可通行区域不建边，不依赖 `walkable=false` 去隐藏错误道路。
- `walkable=false` 只用于必要的显示辅助或 POI 锚定边。
- 每条边必须有对应几何点。
- 曲线距离应由几何点累计计算，而不是手工随意填写。
- 对于透视地图，`distance_m_est_curved` 建议结合实际校园道路距离校准，不建议只使用单一像素比例。

## 8. 推荐实施流程

建议后续按以下顺序实施：

1. 固定新底图文件尺寸，确认渲染使用的图片路径。
2. 在图像上人工标注道路中心线节点。
3. 按分区生成新的 `nodes.csv`。
4. 沿道路中心线生成 `edge_geometry_points.csv`。
5. 自动计算每条边的 `distance_px`、`distance_px_curved` 和几何点数量。
6. 用人工校准比例生成 `distance_m_est` 和 `distance_m_est_curved`。
7. 生成新的 `edges_curved.csv`。
8. 运行连通性检查，确认无孤立节点、无重复边、无跨水错误连接。
9. 运行典型路线测试，重点验证宿舍区、教学区、体育区、图书馆、东校门、南门之间的路径。
10. 再考虑是否提高容量上限和增加更精细的 POI。

## 9. 结论

新上传地图适合重建一套更真实的校园步行网络，但不适合直接复用旧节点坐标，也不适合用简单全局比例自动生成距离。

推荐方案是：

- 使用新图片作为底图。
- 按道路中心线重新标注约 156 个道路节点。
- 设置约 230 到 250 条可通行边。
- 对湖区、桥梁、宿舍区和教学区使用更密集的曲线几何点。
- 明确禁止穿越建筑、水体、草坪和体育场内部。
- 后续实施前提高节点和几何点容量上限，但不修改 Dijkstra 和 A*。
